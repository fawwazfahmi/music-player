# Music Universe — Phase 0: Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bootable, dockerized Next.js 15 app with PostgreSQL + Prisma, password-gated by a signed cookie, ready for the iPod UI in Phase 1.

**Architecture:** A single Next.js 15 (App Router, TypeScript strict) service runs alongside a PostgreSQL 16 container via `docker compose`. All routes except `/api/login` and `/login` are blocked by `middleware.ts` until a `mu_session` cookie signed with `COOKIE_SECRET` (HMAC-SHA256) is present. The cookie is set by `/api/login` after `bcryptjs.compare(password, APP_PASSWORD_HASH)` succeeds. Prisma owns the schema for every entity the spec defines, but Phase 0 only exercises connectivity; data flows arrive in Phase 1.

**Tech Stack:** Next.js 15 (App Router), TypeScript (strict), Tailwind v4, shadcn/ui, Prisma 5, PostgreSQL 16 + `pg_trgm`, Zustand (installed but unused until Phase 1), Vitest + Testing Library, Zod, bcryptjs, Docker + docker-compose.

**Reference:** Spec at `docs/superpowers/specs/2026-06-04-music-universe-design.md` — especially §2 (architecture), §3 (data model), §8 (folder structure), §9 (env vars).

```ts
const Schema = z.object({
  DATABASE_URL: z.string().url(),
  MUSIC_LIBRARY_PATH: z.string().min(1),
  YT_DLP_PATH: z.string().min(1),
  FFMPEG_PATH: z.string().min(1),

  COOKIE_SECRET: z.string().min(32),

const parsed = Schema.safeParse(process.env);
if (!parsed.success) {
  const issues = parsed.error.issues
    .map((i) => `  - ${i.path.join(".")}: ${i.message}`)
    .join("\n");
  throw new Error(`Invalid environment variables:\n${issues}`);
}

export const env = parsed.data;
```

Create `src/server/auth.ts`:

```ts
import crypto from "node:crypto";
import bcrypt from "bcryptjs";
import { env } from "@/lib/env";

const SEP = ".";

export function signCookie(payload: string): string {
  const b64 = Buffer.from(payload, "utf8").toString("base64url");
  const mac = crypto
    .createHmac("sha256", env.COOKIE_SECRET)
    .update(b64)
    .digest("base64url");
  return `${b64}${SEP}${mac}`;
}

export function verifyCookie(value: string | undefined | null): string | null {
  if (!value) return null;
  const [b64, mac] = value.split(SEP);
  if (!b64 || !mac) return null;
  const expected = crypto
    .createHmac("sha256", env.COOKIE_SECRET)
    .update(b64)
    .digest("base64url");
  const a = Buffer.from(mac);
  const b = Buffer.from(expected);
  if (a.length !== b.length) return null;
  if (!crypto.timingSafeEqual(a, b)) return null;
  try {
    return Buffer.from(b64, "base64url").toString("utf8");
  } catch {
    return null;
  }
}

export async function verifyPassword(plain: string): Promise<boolean> {
  return bcrypt.compare(plain, env.APP_PASSWORD_HASH);
}

export const SESSION_COOKIE_NAME = "mu_session";
export const SESSION_COOKIE_VALUE = "ok"; // single-user; presence + signature is what counts
```

## Task 9: Add login API route + login page

**Files:**
- Create: `src/app/api/login/route.ts`, `src/app/api/logout/route.ts`, `src/app/login/page.tsx`

- [ ] **Step 1: Write the login route**

Create `src/app/api/login/route.ts`:

```ts
import { NextResponse } from "next/server";
import { z } from "zod";
import {
  SESSION_COOKIE_NAME,
  SESSION_COOKIE_VALUE,
  signCookie,
  verifyPassword,
} from "@/server/auth";

const Body = z.object({ password: z.string().min(1) });

export async function POST(req: Request) {
  const json = await req.json().catch(() => null);
  const parsed = Body.safeParse(json);
  if (!parsed.success) {
    return NextResponse.json({ ok: false, error: "bad_request" }, { status: 400 });
  }
  const ok = await verifyPassword(parsed.data.password);
  if (!ok) {
    return NextResponse.json({ ok: false, error: "invalid_password" }, { status: 401 });
  }
  const res = NextResponse.json({ ok: true });
  res.cookies.set({
    name: SESSION_COOKIE_NAME,
    value: signCookie(SESSION_COOKIE_VALUE),
    httpOnly: true,
    sameSite: "lax",
    secure: process.env.NODE_ENV === "production",
    path: "/",
    maxAge: 60 * 60 * 24 * 365, // 1 year
  });
  return res;
}
```

## Task 10: Add password-gate middleware

**Files:**
- Create: `src/middleware.ts`

- [ ] **Step 1: Write the middleware**

Create `src/middleware.ts`:

```ts
import { NextResponse, type NextRequest } from "next/server";
import { SESSION_COOKIE_NAME, verifyCookie, SESSION_COOKIE_VALUE } from "@/server/auth";

const PUBLIC_PATHS = ["/login", "/api/login"];

export function middleware(req: NextRequest) {
  const { pathname } = req.nextUrl;

  // Allow Next internals + public paths
  if (
    pathname.startsWith("/_next") ||
    pathname.startsWith("/favicon") ||
    PUBLIC_PATHS.some((p) => pathname === p || pathname.startsWith(`${p}/`))
  ) {
    return NextResponse.next();
  }

  const cookie = req.cookies.get(SESSION_COOKIE_NAME)?.value;
  const verified = verifyCookie(cookie);
  if (verified !== SESSION_COOKIE_VALUE) {
    const url = req.nextUrl.clone();
    url.pathname = "/login";
    url.searchParams.set("from", pathname);
    return NextResponse.redirect(url);
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/((?!_next/static|_next/image).*)"],
};
```

Note: Next.js middleware runs on the Edge runtime by default, but `@/server/auth` imports `bcryptjs` and `node:crypto`. We need the Node runtime for the middleware. Add this just below the imports:

```ts
export const runtime = "nodejs";
```

So the final middleware file is:


## Self-Review Notes (for the engineer)

- **Spec coverage check:** Phase 0 corresponds to spec §7 Phase 0. It covers env config, Docker, Prisma schema (all entities, not just Phase 0 ones — this avoids schema churn between phases), password-gate middleware, login page. Library scanning, the iPod UI, audio streaming, YT, metadata enrichment, stats, and Wrapped are **explicitly out of scope** for this plan — they belong to Phases 1-6, each with its own plan.
- **What this plan deliberately does NOT do:** No iPod chassis component, no click wheel, no audio engine, no chokidar watcher, no Zustand stores. The login screen is intentionally plain — the iPod-styled login is part of Phase 1's polish, when the design system primitives exist.
- **TDD scope:** Only the pure server utilities (env, auth) have unit tests. The middleware + login flow are verified manually in Task 12 because they're integration concerns; we'll add a Playwright integration test in Phase 1 once the iPod UI exists and there's a meaningful page to test against.
- **Schema commitment:** The full schema lands in Phase 0 even though most tables are unused until later phases. Reason: schema migrations are cheap once, painful later when there's real data. Better to take the migration hit now.
- **Why no `app` service in `docker-compose.yml` yet:** Phase 0-5 development happens against `pnpm dev` for fast HMR. The production app container target lands in Phase 6 alongside Cloudflare Tunnel docs.
